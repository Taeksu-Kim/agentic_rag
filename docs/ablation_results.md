## 해석 (2026-07-18 분석)

**리랭크 축**: 0.6B 크로스인코더가 스위트스팟. 9B LLM 리랭크가 MRR +2pp 더 높지만
비용이 ~30배(0.16s vs 4.3s 순차). sparse 단독은 리랭커로도 회생 불가(1단계가 못
가져온 문서는 리랭커가 못 살린다).

**검색 축**: 리랭크 없이는 dense가 hybrid를 이긴다 — 형태소 분석 없는 한국어
BM25(R@8 0.238)가 RRF를 오염. 리랭커를 얹으면 hybrid가 회복(후보 풀 확장 기여,
노이즈는 리랭커가 필터). dense+ce와 hybrid+ce는 사실상 동급.

**에이전트 축 (정직한 네거티브 결과)**: react 반복이 이 평가셋에선 1-shot
hybrid+ce를 넘지 못한다 (R@8 0.515 vs 0.572 @s6). 정밀도 각도(P@1/P@3)로 봐도
동일하게 열세. 원인 두 가지:
1. **law_names 필터 오용**: 9B가 존재하지 않는 법령명 표기로 필터를 걸어 0건 →
   s2에선 만회 불가(빈손 17%). 스텝 예산이 늘면 회복(빈손 20→12→9개)하지만
   비용이 latency 25→112s로 폭증.
2. **평가셋 특성**: 질의회시 질문은 이미 법률 용어로 정제된 문장이라 에이전트의
   핵심 무기인 리라이팅의 이득 여지가 작다. 일상어 질문(예: "육아휴직 때 연차
   나와요?" — 라이브 검증에서 리라이팅으로 top-30 밖→1위)에서는 반대 결과가
   예상됨 → 일상어 변형 평가셋이 후속 과제.

**청킹 불필요 재확인 (데이터 근거)**: 정답 조문 길이별 top-8 회수율(hybrid+ce)이
~500자 0.445 < 500~1000자 0.587 < 1000~2000자 0.606 — miss는 **긴 조문이 아니라
짧은 조문**(정의·준용 조항류, 저정보)에 몰려 있다. 임베딩 절단이 문제가 아니므로
항 단위 청킹은 이득이 없다. (2000자+ 라벨은 n=6뿐이라 통계 무의미.)

**재현**: `PYTHONPATH=. python scripts/run_ablations.py retrieval|agent|report`
(재개 가능; per-query 결과는 data/eval/ablation/*.jsonl). 병렬(--workers 3) 실행
시 latency는 서버 큐잉 포함 참고치(순차 실측: hybrid+ce 0.16s, hybrid+llm 4.3s).
